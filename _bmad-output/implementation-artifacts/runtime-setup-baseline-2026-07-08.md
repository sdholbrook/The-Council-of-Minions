# Runtime and Setup Baseline - 2026-07-08

## Purpose

Close the implementation-readiness gap called out in the BMAD readiness report: the repo had no application runtime manifest, no local run command, no validation command, no packaging boundary, and no minimal check script.

This artifact establishes the first implementation baseline without authorizing tenant writes.

## Current Repo Reality

No starter application runtime manifest exists in the repo:

- No `package.json`
- No `requirements.txt`
- No `pyproject.toml`
- No `go.mod`
- No `Cargo.toml`
- No `.sln`, `.csproj`, or `.cdsproj`

Current implementation artifacts are planning/runtime-adjacent:

- Dataverse schema manifest JSON.
- Dataverse manifest validator.
- Dataverse deployment dry-run plan.
- Dataverse read-only preflight script.
- BMAD sprint status YAML.
- Local validation script.

## Runtime Decision

### MVP Runtime

The practical MVP runtime is:

1. **Dataverse operational store** for Source Records, Work Items, Receipts, Graph Entities/Edges, Memory Candidates, Approved Instructions, Skills, Minions, Briefs, Tenant Evidence, and Platform Evaluations.
2. **Model-driven app** named `Council Queue` as the first review surface for Council Queue and Minion Brief behavior.
3. **Repo-managed manifest and exported solution artifacts** as the source-controlled implementation representation.
4. **BMAD workflow artifacts** as the planning and story-control harness, not product runtime.

This runtime decision is ready for Doug approval. It still requires tenant validation and explicit Dataverse write approval before execution.

### Non-Runtime Boundaries

The following are not MVP runtime dependencies:

- No custom web app.
- No local dev server.
- No published Copilot Studio agent.
- No Power Automate flow.
- No app registration.
- No Fabric workspace, Fabric IQ ontology, or Fabric Graph object.
- No Power Apps MCP agent feed dependency for the first proof.
- No Outlook/Graph live read dependency for the first mock/manual slice.

These can be evaluated later through Platform Evaluation records.

## Local Run Command

There is no application to run locally yet. The local "run" command is validation of the implementation packet:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File _bmad-output\implementation-artifacts\council-mvp-local-validate.ps1
```

Expected success marker:

```text
COUNCIL_MVP_LOCAL_VALIDATE_OK
```

## Validation Scope

The local validation command checks:

1. Git whitespace with `git diff --check`.
2. BMAD config resolution.
3. Dataverse schema manifest validity.
4. Dataverse deployment plan remains dry-run only.
5. `epics.md` has no template placeholders.
6. `sprint-status.yaml` parses and contains 5 epics, 25 stories, and 5 retrospectives.
7. UX spines exist and contain required DESIGN/EXPERIENCE sections.
8. Stale BMAD paused-gate references are absent.

It does not validate:

- Live Dataverse connectivity.
- Tenant identity.
- Dataverse write permission.
- Model-driven app rendering in tenant.
- Actual solution export/import.
- Outlook/Graph read path.
- DLP, sensitivity, retention, licensing, or capacity.

Those remain tenant gates.
